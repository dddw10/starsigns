第一课：Dify 工作流核心概念与自动化思维

目标：明确 Chatflow 与 Workflow 的边界，理清数据流的传递规则

1. 前端开发者的自动化视角

作为前端开发者，我们习惯于编写 function。一个函数包含：输入 (Input) ➔ 逻辑处理 (Process) ➔ 输出 (Output)。同样，AI 软件中的工作流本质上也是一个可视化的复杂函数。

在 Dify 中，有两类应用极易混淆：

  • Chatflow（聊天流）：为对话设计，自带 Session、上下文管理和历史记录，适合做聊天机器人。
  • Workflow（工作流）：为任务设计，单向执行，输入确定，输出确定，没有隐式的上下文干扰。非常适合作为 API 服务，用来跑通自动化流程。
开发原则：进行软件开发自动化（如代码生成、规范检查）时，永远优先选择 Workflow。

2. 变量：工作流中的“数据链路”

在前端，我们用变量在函数间传递数据。Dify 工作流中也是如此。Dify 中的变量传递遵循以下规则：

  1. 起始输入 sys.query 或 Start 节点变量：工作流的入口参数，类似于 API 请求的 body。
  2. 节点级输出：每个节点执行完毕后，会输出一个对象（例如 LLM 节点输出的 text）。
  3. 链式引用：后置节点通过 {{#nodeName.variableName#}} 语法，精确读取前置节点的输出。这与我们在 JS 中使用 nodeName.variableName 极其相似。

例如，若我们有一个名为 translate_code 的节点，它的输出是翻译后的代码，那么在下一个节点中，我们通过 {{#translate_code.text#}} 来获取它。
